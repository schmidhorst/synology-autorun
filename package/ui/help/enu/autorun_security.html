Security

Script location:

There are two fundamentally different places where the script started by the autorun package can be placed:

  • On the external storage devices:
    • Only the storage devices prepared with the script in this way will start the action. All other storages that may be plugged in will be ignored.
    • If the available security option is not used, then anyone who has access to the disk station can do arbitrary things by plugging in a prepared USB stick. E.g. delete the whole disk station and have free access to the whole network.
  • Internal in DSM:
    • Every time a storage device is plugged in, the script is started. The script itself may have to check whether it should perform its actions on the current storage device.
    • The available security option will not be necessary most of the time.

In addition to the primary script mentioned above, a secondary script, which is executed after the storage device has been ejected, is also optionally possible. This second script can naturally only be located internally on the DSM. Access to the ejected memory is no longer possible at the time when it is to be started.

All scripts run with root permission. Internally placed scripts can therefore be located almost anywhere in the DSM file system. Users without administrator rights should not have access, at least not write access!

Security option:

Script execution can be restricted to scripts previously registered using their checksum (SHA256 hash). Other scripts will not be executed.
There are two different approaches:

  1. In the settings during autorun package installation, the "1 first script allowed" setting is selected, and the checksum of the first script after installation is determined and registered. Only this script and unchanged copies of it are executed. The security automatically arms with the first script after the initial, and if necessary also after a re-installation. If the script should ever need a change, then afterwards make again an autorun package re-installation with this setting "1 first script allowed". The old registration is deleted and the new script checksum is registered afterwards as the only permissible one.
  2. During the package installation first "2 Add any hash" is selected. Now all the external storages which have different scripts need to be plugged in one after the other times. So that the needed checksums can be registered and added. After that the installation need to be run again and the value have to be changed to "3 Only previously registered". From then on the security is armed.
Procedure B can also be used if A was previously in use and an additional checksum, e.g. for a new script variant, needs to be added in parallel to the previously registered script variant. "
